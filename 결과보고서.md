### 기관선정
※ 주요 공공기관/데이터 선정 근거

#### 1) 주관 핵심기관: 국토안전관리원(중심 데이터)
- 지반침하 이력, GPR 탐사, 시설물 안전관리 데이터가 서비스 핵심 위험도 산정축이다.
- 점수 산정에서 핵심 가중치(과거사고, GPR, 시설물)는 국토안전관리원 성격 데이터로 구성된다.
- 공모 주제의 “공공데이터 기반 사회문제 해결”과 직접 정합된다.

#### 2) 연계기관 및 공공데이터
- 기상청: 최근 강우량 기반 단기 위험 상승 반영.
- 공공데이터포털/지하수 공공데이터: 지하수 변동성 기반 지반 불안정 반영.
- 지자체/공간환경 데이터: 건물 밀집도, 도로 밀집도, 토지이용 등 지역 특성 반영.

#### 3) 선정 타당성(점수 대응)
- 단일 데이터 조회가 아닌, **핵심(국토안전) + 보조(기상/지하수/환경)** 융합 구조.
- 위험 점수, 지도, 그래프, 설명 리포트까지 이어지는 서비스 완결성 확보.
- 공사 데이터는 보조 변수로만 제한하여 서비스 정체성(지반침하 안전관리) 유지.

### 1) AI 활용
※ 발굴한 서비스에 활용한 AI(알고리즘/분석 방식) 구체 제시

#### 1-1. AI 적용 구조(현재 구현)
- 분석 AI: 규칙기반 위험도 산정 엔진
  - `0~100` 점수 산정, `낮음/보통/높음/매우 높음` 등급화.
  - 항목별 기여도 분해 제공(과거사고, GPR, 시설물, 강우, 지하수, 환경, 공사보조).
- 설명 AI: Gemini 기반 자연어 리포트
  - 분석 결과 + 기여도 + feature를 프롬프트에 투입.
  - 실패 시 템플릿 보고서 fallback 제공(운영 안정성 확보).

#### 1-2. 위험도 산정 알고리즘(실구현 수식)
- 과거사고: `past_sinkhole_count * 8` (최대 30)
- GPR: `gpr_detected_count * 12` (최대 30)
- 시설물: `facility_aging_score * 0.25` (최대 15)
- 강우: `rainfall_score` (최대 10)
- 지하수: `groundwater_score` (최대 8)
- 환경: `environment_score` (최대 6)
- 공사보조: `construction_score * 0.2` (최대 4)
- 등급 기준:
  - 0~29 낮음
  - 30~59 보통
  - 60~79 높음
  - 80~100 매우 높음

#### 1-3. 설명형 AI 프롬프트(실구현)
- 입력:
  - 지역 정보(지역명/행정구역/좌표)
  - 분석일/위험 점수/위험 등급
  - 규칙기반 기여도(7항목)
  - 주요 feature 값
- 출력 요구:
  - 위험 수준 요약
  - 핵심 원인 2~4개
  - 점검/모니터링 권고
  - 불확실성 및 추가 확인 사항

#### 1-4. 고도화 가능성(문서+환경 준비)
- 문서상 목표: `규칙기반 + XGBoost + SHAP + 생성형 AI` 구조.
- `requirements.txt`에 `xgboost`, `shap`, `joblib` 포함되어 학습형 모델 확장 기반 확보.
- 현재 버전은 공모전 시연 안정성을 위해 규칙기반 중심으로 동작.

### 2) 활용 데이터
※ 발굴한 서비스에 활용한 데이터 종류/특성/활용 방식 구체 작성

#### 2-1. 데이터 종류(실DB 기준)
- 기준 테이블
  - `regions`
- 원천 데이터
  - `sinkhole_history`(과거 사고 이력)
  - `gpr_inspection`(공동 탐지)
  - `facility_safety`(노후도)
  - `weather_data`(강우/기온/습도)
  - `groundwater_data`(수위/변동)
  - `environment_features`(건물/도로 밀집도, 토지이용)
  - `construction_events`(보조 요인)
- 가공/결과 데이터
  - `feature_dataset`
  - `risk_analysis_result`
  - `ai_report`(리포트 캐싱)

#### 2-2. 데이터 처리 방식
- 분석 단위: `region_id + analysis_date`
- 흐름:
  - 원천 데이터 적재
  - 지역 단위 집계
  - feature 점수화
  - 위험도 계산
  - 결과 저장/재사용
- 주요 feature(실구현):
  - `past_sinkhole_count`
  - `gpr_detected_count`
  - `facility_aging_score`
  - `rainfall_score`(최근 7일 누적강우 기반)
  - `groundwater_score`(최근 7일 변동성 기반)
  - `environment_score`(건물/도로 밀집도 기반)
  - `construction_score`(보조)

#### 2-3. 정량 현황(2026-04-20 기준 실데이터)
- regions: 5건
- sinkhole_history: 12건
- gpr_inspection: 7건
- facility_safety: 5건
- weather_data: 150건
- groundwater_data: 150건
- environment_features: 5건
- construction_events: 4건
- feature_dataset: 5건
- risk_analysis_result: 5건

#### 2-4. 데이터 활용의 강점
- 공공데이터를 점수/등급/우선순위로 변환해 의사결정에 직접 사용 가능.
- 지도·차트·리포트의 동일 근거 데이터 사용으로 설명 일관성 확보.
- 공사 변수는 보조 가중치로 제한해 과대해석 리스크 최소화.

### 3) 필요성 및 목적
※ 제안 내용의 필요성과 추진 목적

#### 3-1. 필요성
- 지반침하 대응은 사고 이후 조치에 치우치기 쉬워 선제 대응 체계가 필요하다.
- 점검 대상 선정이 경험 중심일 경우, 데이터 기반 우선순위 판단이 약해진다.
- 공공데이터가 분산되어 있어 현장에서 “원인 + 조치”까지 한 번에 보기 어렵다.

#### 3-2. 서비스 목적
- 지역별 위험을 정량화해 점검 우선순위를 제시한다.
- “어디가 위험한가”뿐 아니라 “왜 위험한가”를 함께 설명한다.
- 분석결과를 지도/그래프/AI보고서로 연결해 실무 의사결정을 지원한다.

#### 3-3. 기대 사용자
- 지자체 실무자: 우선 점검 대상 선정 및 자원 배분.
- 현장 점검팀: 점검 전 사유 확인 및 계획 수립.
- 정책/관리자: 지역별 위험 추이 기반 정책 우선순위 설정.

#### 5-2. 예상 질의응답(PT 점수 대비)
1. 왜 생성형 AI를 예측에 쓰지 않았나?
   - 예측은 정형 데이터 모델이 담당, 생성형 AI는 설명에만 사용해 신뢰성과 통제성을 확보했다.
2. 공사데이터가 중심이 아닌 이유는?
   - 문서 원칙대로 공사 변수는 보조 가중치(최대 4점)로 제한했다.
3. AI API 실패 시 서비스가 중단되지 않나?
   - 템플릿 보고서 fallback으로 분석 결과 제공을 유지한다.
4. 실무에서 바로 쓸 수 있나?
   - 우선순위 표, 기여도, PDF 보고서까지 이미 연결돼 의사결정 문서로 바로 활용 가능하다.
5. 확장 계획은?
   - XGBoost/SHAP 기반 학습형 모델과 PostgreSQL/PostGIS로 고도화한다.

#### 5-3. 향후 고도화 계획
- 모델 고도화:
  - XGBoost 학습모델 적용, SHAP 기반 개별 설명 제공.
- 데이터 고도화:
  - 실시간/주기 갱신 파이프라인 자동화.
- 운영 고도화:
