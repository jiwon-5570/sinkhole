# 03_DB_제작_가이드

## 1. 목적

이 문서는 `Project/backend/db/schema.sql`에 정의된 SQLite 스키마와 운영 DB 설계 기준을 설명한다. 현재 DB는 원본 공공데이터, 정규화 데이터, feature 캐시, 위험 분석 결과, 리포트/모니터링 관련 데이터를 저장한다.

## 2. DB 기본 원칙

- 기본 DB는 SQLite이며 경로는 `SINKHOLE_DB_PATH`로 변경할 수 있다.
- 서버 시작 시 `SINKHOLE_APPLY_SCHEMA_ON_START=1`이면 스키마가 자동 적용된다.
- 스키마 적용은 `CREATE TABLE IF NOT EXISTS`와 보조 `ALTER TABLE` 방식으로 기존 운영 DB를 보존한다.
- 원본 데이터와 분석 결과를 분리한다.
- 지역 분석은 `regions.region_id`, 도로 분석은 `road_segments.road_id`를 기준으로 연결한다.
- 데모 seed 데이터는 사용하지 않는다.

## 4. 핵심 관계

```text
regions
├─ sinkhole_history
├─ gpr_inspection
├─ facility_* 
├─ weather_data
├─ groundwater_data
├─ environment_features
├─ construction_events
├─ feature_dataset
├─ risk_analysis_result
└─ road_segments
   ├─ road_* source tables
   ├─ road_feature_dataset
   └─ road_risk_analysis_result
```

원본 데이터 수집 테이블은 `source_name`, `source_record_id`, `payload_hash`를 사용해 중복과 출처를 관리한다.

## 5. 주요 테이블 설명

### 5.1 `regions`

분석의 기본 단위다.

주요 컬럼:

- `region_id`
- `region_name`
- `region_type`
- `latitude`
- `longitude`
- `sido`
- `sigungu`
- `geom`

현재 서비스는 공공 지층 데이터 기반 서울/수도권 분석 지점을 생성해 사용할 수 있다.

### 5.2 `road_segments`

도로 단위 분석 확장을 위한 테이블이다. 현재 운영 DB에는 도로 데이터가 없을 수 있으므로 API와 UI는 빈 목록을 정상 처리해야 한다.

주요 컬럼:

- `road_id`
- `region_id`
- `road_name`
- `center_lat`
- `center_lon`
- `length_m`
- `geometry`

### 5.3 `feature_dataset`

지역 단위 위험도 계산 입력 캐시다.

Primary key:

```text
(region_id, analysis_date)
```

주요 feature:

- `past_sinkhole_count`
- `gpr_detected_count`
- `facility_aging_score`
- `rainfall_score`
- `groundwater_score`
- `environment_score`
- `construction_score`

### 5.4 `risk_analysis_result`

지역 단위 분석 결과다.

주요 컬럼:

- `region_id`
- `analysis_date`
- `total_risk_score`
- `risk_level`
- `priority_rank`

분석 실행 시 같은 날짜/지역의 기존 결과가 있으면 갱신하고, 전체 우선순위를 다시 계산한다.

### 5.5 `construction_events`

공사 영향 feature의 원천이다. 서울시 도로굴착 로컬 파일, 서울 열린데이터, 지하안전/건축 인허가 데이터가 반영될 수 있다.

주요 컬럼:

- `region_id`
- `construction_type`
- `start_date`
- `scale_score`
- `source_name`
- `source_record_id`
- `address`
- `latitude`
- `longitude`

### 5.6 `molit_ground_boreholes`, `molit_ground_layers`

시추공과 지층 정보를 저장한다. 지하수 및 지층/환경 보정에 사용된다.

시추공 주요 컬럼:

- `borehole_code`
- `latitude`, `longitude`
- `raw_x`, `raw_y`
- `coordinate_crs`
- `groundwater_level_m`
- `total_depth_m`

지층 주요 컬럼:

- `borehole_code`
- `top_depth_m`
- `bottom_depth_m`
- `thickness_m`
- `layer_name`
- `soil_class`
- `n_value`

### 5.7 `monitoring_points`

사용자가 등록한 위치 기반 모니터링 지점을 저장한다.

주요 컬럼:

- `name`
- `address`
- `latitude`
- `longitude`
- `risk_score`
- `risk_level`
- `last_checked_at`
- `last_error`
- `active`

## 6. DB 초기화

```powershell
cd .\Project\backend
python .\scripts\init_db.py
```

이 명령은 스키마만 적용하며 데모 데이터를 삽입하지 않는다.

## 7. 데이터 적재와 캐시 무효화

공공데이터나 로컬 파일이 변경되면 다음 캐시를 삭제한다.

- `feature_dataset`
- `road_feature_dataset`

이후 오늘 날짜 기준으로 `analyze_region()`과 `analyze_road()`가 다시 실행되어 결과 테이블을 갱신한다.

## 8. 인덱스 기준

현재 스키마는 다음 조회 패턴을 고려해 인덱스를 둔다.

- 지역별 사고 이력 조회
- 날짜별 강우 조회
- 지역/분석일별 위험 결과 조회
- 도로/분석일별 위험 결과 조회
- 원본 수집 레코드 출처별 조회
- 지층/시추공 좌표 기반 조회

운영 중 느린 쿼리가 발견되면 `EXPLAIN QUERY PLAN`으로 확인한 뒤 인덱스를 추가한다.

## 9. 마이그레이션 주의사항

- 운영 DB 파일은 대용량일 수 있으므로 스키마 변경 전 백업한다.
- 기존 사용자 데이터와 공공데이터 적재 결과를 삭제하지 않는다.
- `DELETE`가 필요한 재정규화 작업은 출처가 명확한 테이블만 대상으로 한다.
- 새 컬럼은 가능한 nullable로 추가하고 코드에서 fallback을 둔다.

## 10. 점검 쿼리

```sql
SELECT COUNT(*) FROM regions;
SELECT COUNT(*) FROM construction_events;
SELECT COUNT(*) FROM molit_ground_boreholes;
SELECT COUNT(*) FROM molit_ground_layers;
SELECT * FROM risk_analysis_result ORDER BY analysis_date DESC, priority_rank ASC LIMIT 10;
SELECT * FROM local_file_import_state ORDER BY imported_at DESC;
```

## 11. 확장 방향

- 운영 규모가 커지면 PostgreSQL/PostGIS로 이전한다.
- 도로 단위 분석을 본격화하려면 `road_segments`를 실제 도로망 데이터로 채운다.
- 시간 기반 ML 학습을 위해 feature snapshot과 label 테이블을 별도로 둘 수 있다.
- 원본 수집 run별 비교를 위해 `public_data_collection_runs` 상세 결과 테이블을 추가할 수 있다.
