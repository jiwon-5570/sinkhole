# 국토안전관리 프로젝트 DB 제작 가이드

## 1. 문서 목적

이 문서는 국토안전관리 프로젝트의 데이터베이스 설계를 정의한다.

본 가이드는 다음을 목표로 한다.

1. 공공데이터 및 내부 데이터를 저장할 DB 구조 설계
2. 지역 기반 데이터 통합 구조 정의
3. 분석 및 모델 학습에 필요한 Feature 구조 정의
4. 결과 저장 및 API 연동을 위한 테이블 설계
5. 확장 가능한 DB 구조 구축

---

## 2. DB 설계 기본 원칙

### 2.1 지역 기반 설계
모든 데이터는 반드시 **지역(region)** 기준으로 연결된다.

→ 모든 테이블은 `region_id`를 통해 연결

---

### 2.2 원천 데이터 / 분석 데이터 분리

데이터는 3단계로 나눈다.

1. 원천 데이터 (Raw Data)
2. 가공 데이터 (Feature Data)
3. 분석 결과 (Result Data)

---

### 2.3 확장 가능 구조

- 테이블 간 관계는 느슨하게 유지
- 새로운 데이터 추가 가능하도록 설계
- API 연동을 고려한 구조

---

## 3. 전체 DB 구조

regions

4. 핵심 테이블 정의
4.1 regions (기준 테이블)
목적
모든 데이터의 기준
지도 시각화 기준
구조
CREATE TABLE regions (
    region_id INTEGER PRIMARY KEY,
    region_name TEXT,
    region_type TEXT,
    latitude REAL,
    longitude REAL,
    sido TEXT,
    sigungu TEXT,
    geom TEXT
);
4.2 sinkhole_history
목적
과거 지반침하 이력 저장
CREATE TABLE sinkhole_history (
    id INTEGER PRIMARY KEY,
    region_id INTEGER,
    occurrence_date TEXT,
    cause_type TEXT,
    damage_scale REAL,
    FOREIGN KEY (region_id) REFERENCES regions(region_id)
);
4.3 gpr_inspection
목적
GPR 탐사 및 공동 탐지 데이터
CREATE TABLE gpr_inspection (
    id INTEGER PRIMARY KEY,
    region_id INTEGER,
    inspection_date TEXT,
    cavity_detected INTEGER,
    cavity_count INTEGER,
    depth_estimate REAL,
    FOREIGN KEY (region_id) REFERENCES regions(region_id)
);
4.4 facility_safety
목적
시설물 노후도 및 안전성
CREATE TABLE facility_safety (
    id INTEGER PRIMARY KEY,
    region_id INTEGER,
    facility_type TEXT,
    aging_score REAL,
    inspection_count INTEGER,
    FOREIGN KEY (region_id) REFERENCES regions(region_id)
);
4.5 weather_data
목적
기상 데이터 저장
CREATE TABLE weather_data (
    id INTEGER PRIMARY KEY,
    region_id INTEGER,
    record_date TEXT,
    rainfall REAL,
    temperature REAL,
    humidity REAL,
    FOREIGN KEY (region_id) REFERENCES regions(region_id)
);
4.6 groundwater_data
목적
지하수위 데이터 저장
CREATE TABLE groundwater_data (
    id INTEGER PRIMARY KEY,
    region_id INTEGER,
    record_date TEXT,
    groundwater_level REAL,
    variation REAL,
    FOREIGN KEY (region_id) REFERENCES regions(region_id)
);
4.7 environment_features
목적
주변 환경 데이터
CREATE TABLE environment_features (
    id INTEGER PRIMARY KEY,
    region_id INTEGER,
    building_density REAL,
    road_density REAL,
    land_use_type TEXT,
    FOREIGN KEY (region_id) REFERENCES regions(region_id)
);
4.8 construction_events
목적
공사 데이터 (보조)
CREATE TABLE construction_events (
    id INTEGER PRIMARY KEY,
    region_id INTEGER,
    construction_type TEXT,
    start_date TEXT,
    scale_score REAL,
    FOREIGN KEY (region_id) REFERENCES regions(region_id)
);
4.9 feature_dataset (핵심)
목적
모델 입력 데이터
CREATE TABLE feature_dataset (
    region_id INTEGER,
    analysis_date TEXT,
    past_sinkhole_count INTEGER,
    gpr_detected_count INTEGER,
    facility_aging_score REAL,
    rainfall_score REAL,
    groundwater_score REAL,
    environment_score REAL,
    construction_score REAL,
    PRIMARY KEY (region_id, analysis_date)
);
4.10 risk_analysis_result
목적
분석 결과 저장
CREATE TABLE risk_analysis_result (
    id INTEGER PRIMARY KEY,
    region_id INTEGER,
    analysis_date TEXT,
    total_risk_score REAL,
    risk_level TEXT,
    priority_rank INTEGER,
    FOREIGN KEY (region_id) REFERENCES regions(region_id)
);

6. 데이터 처리 흐름
Raw Data 저장
→ 지역 매핑
→ Feature 생성
→ feature_dataset 저장
→ 모델 실행
→ 결과 저장
7. 인덱스 전략
CREATE INDEX idx_region ON sinkhole_history(region_id);
CREATE INDEX idx_date ON weather_data(record_date);
8. 성능 고려사항
region_id 기반 조회 최적화
시간 기준 조회 인덱스 추가
대용량 데이터는 분할 저장
9. API 연동 고려

API에서 자주 사용하는 테이블:

regions
risk_analysis_result
feature_dataset
10. 최종 요약

이 DB 구조는 다음 3단계를 기반으로 한다.

원천 데이터 저장
Feature 생성
분석 결과 저장

즉, DB는 단순 저장소가 아니라
→ 분석과 서비스 출력을 연결하는 핵심 구조이다.
