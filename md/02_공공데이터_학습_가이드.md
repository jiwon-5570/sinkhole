# 02_공공데이터_학습_가이드

## 1. 목적

이 문서는 싱크홀 위험도 대시보드가 사용하는 공공데이터의 수집, 정규화, DB 반영, feature 생성 기준을 설명한다. 현재 서비스는 데모 seed 데이터를 사용하지 않고, 승인된 공공데이터 API와 사용자가 제공한 로컬 원본 파일을 기준으로 분석한다.

## 2. 데이터 운영 원칙

- 운영 분석에는 임의 seed 데이터를 섞지 않는다.
- 공공데이터 API 키와 민감 설정은 `Project/backend/.env`에만 저장한다.
- 원본 데이터와 정규화 데이터는 분리한다.
- API 수집 실패 시 기존 DB와 로컬 파일을 기준으로 대시보드가 계속 동작해야 한다.
- 공사 데이터는 보조 위험 요인으로 반영하며 단독으로 위험 판단을 확정하지 않는다.
- 데이터가 부족한 지표는 fallback 또는 0점 처리 기준을 명확히 둔다.

## 5. 수집 흐름

```text
API 또는 로컬 파일
-> raw_source_records 또는 파일 상태 기록
-> 정규화 테이블 반영
-> feature_dataset / road_feature_dataset 캐시 삭제
-> 오늘 날짜 위험도 재분석
-> risk_analysis_result 저장
-> UI 갱신
```

공공데이터 수집은 `/api/public-data/refresh` 또는 서버 시작 스케줄러에서 실행된다. 서울 열린데이터만 별도로 수집하려면 `/api/public-data/seoul/import`를 사용한다.

## 6. 로컬 도로굴착 파일 처리

지원 확장자:

- `.csv`
- `.tsv`
- `.txt`
- `.xlsx`
- `.xls`

처리 기준:

- 파일 변경 여부는 `local_file_import_state`의 크기와 수정 시각으로 판단한다.
- 삭제된 파일의 기존 행은 DB에서도 제거한다.
- 착공일/준공일이 최근 공사 기준에 맞는 행만 반영한다.
- 주소 또는 좌표로 가장 가까운 `regions`와 매칭한다.
- 도로 세그먼트가 있으면 `road_construction_events`에도 반영한다.
- 변경 발생 시 feature 캐시를 삭제한다.

수동 적재:

```powershell
cd .\Project\backend
python .\scripts\import_seoul_road_excavation.py
```

## 7. 지층/시추 데이터 처리

지층 CSV는 `scripts/import_molit_ground_data.py`로 적재한다. 시추공 데이터는 기본적으로 OpenAPI 수집을 사용하고, API 장애 시 `molit_boreholes` 폴더의 CSV를 보조로 적재할 수 있다.

```powershell
cd .\Project\backend
python .\scripts\import_molit_ground_data.py
```

좌표 변환 기준은 기본 `EPSG:5186`이며 `.env`에서 변경할 수 있다.

## 8. 정규화 기준

- 날짜는 `YYYY-MM-DD` 또는 ISO datetime으로 저장한다.
- 좌표는 위도/경도 WGS84 기준으로 저장한다.
- 원본 payload는 필요 시 JSON 문자열로 보존한다.
- 원본 식별자는 `source_name`, `source_record_id` 조합으로 관리한다.
- 같은 원본이 반복 수집되어도 중복 삽입되지 않도록 unique 기준을 둔다.

## 11. 학습 데이터 확장 방향

현재 운영 점수는 규칙 기반이다. 향후 ML 모델을 학습하려면 다음 조건을 추가로 만족해야 한다.

- 충분한 기간의 침하 사고 label 확보
- 사고 미발생 지점의 negative sample 정의
- 시간 기준 train/test 분리
- 지역 기준 leakage 방지
- 공사/강우/지하수 데이터의 시점 정렬
- 모델 예측값과 규칙 점수의 혼합 기준 정의

## 12. 주의사항

- 데이터 수집 성공이 곧 위험도 정확도를 보장하지 않는다.
- 과거 이력은 반복 취약성을 보여주는 참고 지표이며 현재 사고 발생을 의미하지 않는다.
- API 키가 없는 환경에서는 로컬 파일과 기존 DB를 기준으로만 동작한다.
- 공공데이터 응답 필드명은 변경될 수 있으므로 alias 기반 매핑을 유지한다.
