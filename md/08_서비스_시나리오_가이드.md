# 08_서비스_시나리오_가이드

## 1. 목적

이 문서는 싱크홀 위험도 대시보드의 실제 사용 시나리오와 발표/시연 흐름을 정리한다. 현재 서비스는 지역 위험 분석, 위치 직접 분석, What-If 시뮬레이션, AI 설명, PDF 리포트를 중심으로 동작한다.

## 2. 주요 사용자

| 사용자 | 목적 |
| --- | --- |
| 지자체/시설 담당자 | 우선 점검 대상과 위험 근거 확인 |
| 현장 관리자 | 특정 주소 또는 좌표의 주변 위험 정보 확인 |
| 발표/평가자 | 데이터 수집부터 분석, 설명, 리포트까지 전체 흐름 확인 |
| 개발/운영자 | 데이터 적재 상태와 API 동작 확인 |

## 3. 기본 대시보드 시나리오

1. `Run_Sinkhole.cmd`로 서버 실행
2. `http://127.0.0.1:5000` 접속
3. 핵심 지표 카드에서 평균 위험도, 분석 지역 수, 높은 위험 지역 수 확인
4. TOP 5에서 우선 점검 대상 확인
5. 지도에서 위험도 색상 확인
6. 선택 지역의 점수와 기여 요인 확인
7. 과거 이력과 원인 분포 확인
8. AI 원인 설명 카드 확인

핵심 메시지:

```text
점수만 보여주는 대시보드가 아니라, 점수 산정 근거와 관리 우선순위를 함께 보여준다.
```

## 4. 지역 위험 분석 시나리오

1. 분석 단위 `지역` 선택
2. 지역 선택
3. 분석일 선택
4. `위험도 분석` 클릭
5. 결과 패널에서 위험 점수와 등급 확인
6. 기여 요인 막대에서 위험 원인 확인
7. 위험도 추이와 지반침하 원인 분포 확인
8. 필요 시 AI Chat에 "왜 이 지역이 위험해?" 질문

성공 기준:

- 분석 결과가 0~100 점수로 표시된다.
- 등급과 우선순위가 표시된다.
- breakdown 값이 화면과 리포트에서 일관된다.

## 5. 도로 분석 시나리오

1. 분석 단위 `도로` 선택
2. 지역 선택
3. 도로 목록 확인
4. 도로 선택 후 위험도 분석 실행

주의:

현재 운영 DB에는 `road_segments`가 비어 있을 수 있다. 이 경우 빈 목록은 오류가 아니라 도로망 데이터가 아직 적재되지 않은 상태다.

## 6. 위치 직접 분석 시나리오

1. 빠른 메뉴에서 실시간 모드 선택
2. `강남역`, `서울특별시 송파구 ...`, 또는 `37.5, 127.0` 형태 입력
3. `지도에 적용`
4. 지도 위치와 주소 표시 확인
5. 주변 분석 지점 기반 위험 정보 확인
6. 필요 시 현장형 리포트 생성

사용 API:

```text
GET  /api/geocode/search
GET  /api/geocode/reverse
POST /api/commercial/analyze
POST /api/commercial/report
```

## 7. What-If 시뮬레이션 시나리오

1. 사이드바에서 `What-If` 열기
2. 프리셋 선택 또는 수동 조정
3. 추가 강우량과 굴착 조건 입력
4. 관리 조치 적용 선택
5. `시뮬레이션 실행`
6. 기존 점수, 시나리오 점수, 관리 후 점수 비교
7. factor 변화와 추천 조치 확인

예시 시나리오:

```text
최근 7일 강우가 증가하고 인근 대규모 굴착이 발생했을 때,
GPR 정밀조사와 배수 개선을 적용하면 점수가 얼마나 줄어드는지 확인한다.
```

## 8. AI Chat 시나리오

추천 질문:

- 현재 위험도가 가장 높은 곳은 어디야?
- 이 지역이 위험한 이유를 설명해줘.
- 공사 영향이 점수에 어떻게 반영됐어?
- 강우와 지하수 중 어떤 요인이 더 커?
- 관리 우선순위는 어떻게 정하면 돼?

AI 답변 기준:

- DB와 분석 결과에 있는 근거를 우선한다.
- 근거가 부족하면 부족하다고 말한다.
- AI는 점수를 새로 계산하지 않는다.

## 9. PDF 리포트 시나리오

1. 지역 분석 실행
2. `AI 리포트 생성` 클릭
3. 리포트 패널에서 요약 본문 확인
4. PDF 목록에서 생성 파일 확인
5. 선택 후 열기/다운로드/삭제 실행

리포트 포함 내용:

- 분석 대상
- 분석일
- 위험 점수와 등급
- 주요 기여 요인
- 근거 데이터 요약
- 관리 권고
- 차트/표 기반 시각 요약

## 10. 모니터링 지점 시나리오

1. 핵심 지표의 모니터링 카드 클릭
2. 주소 또는 좌표 입력
3. `지점 추가`
4. `전체 갱신`
5. 위험 점수와 마지막 확인 시각 확인
6. 필요 없는 지점은 해제

제약:

- 기본 최대 10개
- 설정값: `SINKHOLE_MONITORING_POINTS_MAX_COUNT`
- 갱신 주기: `SINKHOLE_MONITORING_POINTS_REFRESH_SECONDS`

## 11. 공공데이터 운영 시나리오

### 11.1 로컬 도로굴착 파일 갱신

1. CSV/XLSX 파일을 `Project/backend/data/raw/public/seoul_road_excavation/`에 추가
2. 서버가 30초 주기로 감지
3. `construction_events` 반영
4. feature 캐시 삭제
5. 오늘 날짜 위험도 재계산

수동 실행:

```powershell
cd .\Project\backend
python .\scripts\import_seoul_road_excavation.py
```

### 11.2 공공데이터 API 수집

```text
POST /api/public-data/refresh
```

수집 상태:

```text
GET /api/public-data/status
GET /api/public-data/ground-layers/status
GET /api/public-data/local-construction/status
```

## 12. 발표 시연 추천 순서

1. 대시보드 첫 화면에서 전체 현황 소개
2. TOP 5 중 하나를 선택해 지역 분석 실행
3. 기여 요인과 과거 이력 설명
4. AI Chat으로 "왜 위험한지" 질문
5. What-If에서 강우/굴착 조건 적용
6. 관리 조치 적용 후 점수 변화 설명
7. AI 리포트 생성 및 PDF 확인
8. 공공데이터 파일 적재 위치와 자동 반영 구조 설명

## 13. 장애/예외 시나리오

| 상황 | 기대 동작 |
| --- | --- |
| API 키 없음 | 기존 DB/로컬 파일 기준으로 동작 |
| Google 지도 실패 | 자체 지도 또는 새 창 링크 제공 |
| 도로 데이터 없음 | 도로 목록 빈 상태 표시 |
| 차트 데이터 부족 | 빈 상태 또는 fallback 안내 |
| AI 키 없음 | 로컬 규칙 답변 또는 템플릿 리포트 사용 |
| 공공데이터 수집 실패 | 상태 API에 오류 기록, 앱은 계속 동작 |

## 14. 서비스 메시지

서비스를 설명할 때 다음 문장을 기준으로 한다.

```text
공공데이터와 로컬 원본 파일을 통합해 지반침하 위험도를 산정하고,
점수의 근거와 관리 우선순위를 지도, 차트, AI 설명, PDF 보고서로 제공한다.
```
