# 05_예측_알고리즘_가이드

## 1. 목적

이 문서는 현재 운영 중인 지반침하 위험도 계산 로직을 설명한다. 현재 점수는 ML 모델이 아니라 `app/services/features.py`와 `app/services/risk_scoring.py`에 구현된 규칙 기반 알고리즘으로 산출된다. XGBoost/SHAP은 향후 확장 가능성을 위해 의존성에 포함되어 있으나 운영 점수 계산에는 사용하지 않는다.

## 2. 전체 계산 흐름

```text
분석 대상 선택
-> feature row 조회 또는 생성
-> 지층/시설물 사고 보정 적용
-> factor별 기여 점수 계산
-> total risk score 산출
-> 위험 등급 부여
-> priority_rank 재계산
```

지역 분석:

```text
POST /api/analyze-risk
```

도로 분석:

```text
POST /api/analyze-road-risk
```

## 3. 위험 등급 기준

| 총점 | 등급 |
| ---: | --- |
| 0 이상 30 미만 | 낮음 |
| 30 이상 60 미만 | 보통 |
| 60 이상 80 미만 | 높음 |
| 80 이상 100 이하 | 매우 높음 |

점수는 0~100 범위로 clamp한다.

## 4. 입력 Feature

| Feature | 의미 |
| --- | --- |
| `past_sinkhole_count` | 과거 지반침하 사고 건수 |
| `gpr_detected_count` | GPR/탐사 이상 신호 수 |
| `facility_aging_score` | 시설물 노후/점검/사고 기반 위험 점수 |
| `rainfall_score` | 최근 강우 영향 |
| `groundwater_score` | 지하수위 변동 또는 시추공 지하수 대체 지표 |
| `environment_score` | 건물/도로 밀도, 배수, 지층 취약성 |
| `construction_score` | 굴착/공사 영향 |

## 5. Factor별 배점

현재 `risk_scoring.py` 기준:

| Factor | 입력값 변환 | 최대 기여점 |
| --- | ---: | ---: |
| 과거 지반침하 | `past_sinkhole_count * 8.0` | 25 |
| GPR/탐사 | `gpr_detected_count * 12.0` | 30 |
| 시설물/노후 | `facility_aging_score * 0.25` | 15 |
| 강우 | `rainfall_score` | 10 |
| 지하수 | `groundwater_score` | 7 |
| 지층/환경 | `environment_score` | 7 |
| 공사 영향 | `construction_score * 0.3` | 6 |

총점은 factor 기여점 합계다.

## 6. Feature 생성 기준

### 6.1 과거 지반침하

지역:

```sql
SELECT COUNT(*) FROM sinkhole_history WHERE region_id = ?
```

도로:

```sql
SELECT COUNT(*) FROM road_sinkhole_history WHERE road_id = ?
```

### 6.2 GPR/탐사

지역 GPR:

```sql
SELECT SUM(cavity_count) FROM gpr_inspection WHERE region_id = ?
```

MOLIT 물리탐사 데이터가 있으면 탐사 방식에 따라 보조 신호를 추가한다. `GPR` 또는 레이더성 탐사는 더 높은 보조값을 부여한다.

### 6.3 시설물/노후

다음 값을 조합한다.

- `facility_safety.aging_score`
- `facility_status`의 노후 시설 비율
- `facility_inspection.risk_score`
- `facility_accidents`의 최근 사고 위험 보정

사고 이력은 최신 사고일수록 높은 recency weight를 적용한다.

### 6.4 강우

분석일 기준 최근 7일 강우 합계를 사용한다.

```text
rainfall_score = min(10, rainfall_7d_mm / 10)
```

즉 최근 7일 100mm 이상이면 강우 점수는 최대 10점이다.

### 6.5 지하수

우선 `groundwater_data.variation` 최근 7일 평균을 사용한다.

```text
groundwater_score = min(7, avg_variation * 3.5)
```

관측 자료가 없으면 시추공 기반 대체 점수를 계산한다.

- 분석 대상 주변 1.5km 내 시추공 조회
- 평균 지하수 깊이
- 얕은 지하수 비율
- 주변 시추공 밀도

### 6.6 지층/환경

기본 환경 값:

- 건물 밀도
- 도로 밀도
- 도로 분석의 경우 배수 품질, 경사도

지층 보정:

- 주변 지층 데이터 수
- 연약층/매립층/모래층 등 지층명과 N값
- 두께와 깊이

`ground_layers.py`가 주변 지층을 요약하고 `environment_score`에 추가한다.

### 6.7 공사 영향

지역:

```sql
SELECT MAX(scale_score) FROM construction_events WHERE region_id = ?
```

도로:

```sql
SELECT MAX(scale_score) FROM road_construction_events WHERE road_id = ?
```

최대 20점 입력 feature로 저장되고, 최종 기여점은 `construction_score * 0.3`, 최대 6점이다.

## 7. 우선순위 계산

분석 날짜별로 `risk_analysis_result.total_risk_score`를 내림차순 정렬해 `priority_rank`를 다시 부여한다.

도로 분석은 `road_risk_analysis_result`에서 동일한 방식으로 계산한다.

## 8. What-If 시뮬레이션

`simulation_engine.py`는 실제 DB 결과를 변경하지 않고 시나리오 입력값을 기존 feature에 임시 반영한다.

입력 변수:

- 예측 시간 범위
- 추가 강우량
- 지하수위 변화
- 대규모 공사 여부
- 굴착 깊이
- 공사장 거리
- GPR 이상 신호 수
- 시설물 노후 보정
- 과거 침하 추가 건수
- 지층/환경 취약도 보정
- 관리 조치 적용 여부

관리 조치:

- GPR 정밀조사
- 관로 보수
- 배수 개선
- 공사 통제
- 모니터링 강화

결과에는 다음 항목을 포함한다.

- 기존 점수
- 시나리오 적용 점수
- 관리 조치 후 점수
- factor 변화
- 주요 driver
- 추천 조치
- 데이터 품질 설명

## 9. AI 설명과 알고리즘의 관계

AI는 점수를 직접 산출하지 않는다. AI Chat과 리포트는 다음 데이터를 근거로 설명을 생성한다.

- 분석 결과
- feature 값
- breakdown
- 원본 데이터 근거
- 차트/추이/원인 분포

점수 산정 기준은 항상 `risk_scoring.py`가 우선이다.

## 10. 검증 기준

알고리즘 변경 시 확인할 항목:

- 위험 등급 경계값 유지 여부
- factor 최대점 합계가 100인지 여부
- 동일 feature 입력에 대해 동일 점수가 나오는지 여부
- feature 캐시 삭제 후 재계산이 정상인지 여부
- UI 기여 요인 표시와 PDF 리포트 값이 같은지 여부
- What-If가 운영 DB 결과를 직접 수정하지 않는지 여부

## 11. 향후 ML 확장 방향

ML 모델을 도입하려면 다음을 별도 구현한다.

- 시간별 feature snapshot
- 사고 발생 label 정의
- negative sample 정의
- train/test 시간 분리
- 지역 단위 leakage 방지
- 규칙 점수와 ML 점수 비교 지표
- SHAP 기반 설명 결과를 UI/리포트에 연결

현재 운영에서는 규칙 기반 점수가 더 투명하고 데이터 부족에 강하므로 기본 방식으로 유지한다.
