# 09_모델_학습_파이프라인

## 1. 목적

이 문서는 향후 ML 모델을 도입할 때 필요한 학습 파이프라인 기준을 정리한다. 현재 운영 위험 점수는 규칙 기반이며, 이 문서의 ML 절차는 확장 계획이다.

## 2. 현재 상태

현재 운영:

- feature 생성: `features.py`
- 점수 계산: `risk_scoring.py`
- 시뮬레이션: `simulation_engine.py`
- 설명: `ai_evidence.py`, `reporting.py`, `chat.py`

현재 미운영:

- XGBoost 기반 운영 예측
- SHAP 기반 운영 설명
- 모델 artifact 저장/로드
- 정기 재학습

## 3. ML 도입 조건

ML 모델을 운영 점수에 반영하려면 다음 데이터 조건이 필요하다.

- 충분한 지반침하 발생 label
- 사고 미발생 negative sample
- 시간별 feature snapshot
- 데이터 수집 시점과 사고 발생 시점 정렬
- 지역/도로 기준 중복 및 leakage 제거
- 검증 가능한 성능 지표
- 규칙 점수 대비 개선 효과

## 4. 데이터셋 구조

권장 학습 row:

```text
target_id
target_type
analysis_date
past_sinkhole_count
gpr_detected_count
facility_aging_score
rainfall_score
groundwater_score
environment_score
construction_score
ground_layer_score
facility_accident_score
label
label_window_days
```

`target_type`:

- `region`
- `road`

label 예시:

- 분석일 이후 30일 내 지반침하 발생 여부
- 분석일 이후 90일 내 지반침하 발생 여부
- 사고 규모 기반 regression target

## 5. 파이프라인 단계

```text
원본 데이터 수집
-> 정규화 테이블 구축
-> feature snapshot 생성
-> label 생성
-> train/validation/test 분리
-> 모델 학습
-> 성능 평가
-> SHAP/feature importance 산출
-> 운영 반영 여부 검토
```

## 6. Feature Snapshot

현재 `feature_dataset`은 분석일별 캐시 역할을 한다. ML 학습에는 운영 캐시와 별도의 snapshot 테이블을 권장한다.

예시:

```sql
CREATE TABLE model_feature_snapshot (
    id INTEGER PRIMARY KEY,
    target_type TEXT NOT NULL,
    target_id INTEGER NOT NULL,
    analysis_date TEXT NOT NULL,
    feature_json TEXT NOT NULL,
    label INTEGER,
    label_window_days INTEGER,
    created_at TEXT NOT NULL
);
```

장점:

- 알고리즘 변경 전후 feature 비교 가능
- 학습 데이터 재현 가능
- 운영 캐시 삭제와 무관하게 학습 데이터 보존

## 7. 분할 기준

권장 분할:

- Train: 과거 기간
- Validation: 중간 기간
- Test: 최신 기간

주의:

- 같은 사고 주변 지역이 train/test에 동시에 들어가지 않도록 점검한다.
- 동일 원본 수집 run에서 파생된 데이터 leakage를 확인한다.
- 지역별 데이터 편향이 큰 경우 region group split을 검토한다.

## 9. 평가 지표

분류 모델:

- ROC-AUC
- PR-AUC
- Recall@TopK
- Precision@TopK
- F1 score
- Calibration curve

운영 관점 지표:

- 실제 사고가 TOP 5/10에 포함되는 비율
- 높은 위험 등급의 false positive 관리 가능성
- 점검 우선순위 개선 효과

## 10. 운영 반영 방식

권장 단계:

1. 규칙 기반 점수 유지
2. ML 점수를 별도 보조 지표로 계산
3. 일정 기간 shadow mode로 비교
4. 성능과 설명 가능성이 충분하면 혼합 점수 검토

혼합 점수 예시:

```text
final_score = rule_score * 0.7 + ml_score * 0.3
```

혼합 비율은 성능 평가와 현장 검토 후 결정한다.

## 12. 재학습 기준

재학습이 필요한 경우:

- 신규 사고 label 증가
- 공공데이터 원천 변경
- feature 생성 로직 변경
- 지역/도로 대상 변경
- 성능 모니터링 결과 악화

## 13. 운영 주의사항

- AI 생성 문장은 모델 예측값을 대체하지 않는다.
- 모델 점수는 규칙 점수와 산출 근거가 다를 수 있으므로 UI에 구분 표시한다.
- label이 부족한 초기 단계에서는 규칙 기반 점수를 유지한다.
- 모델이 특정 지역에 과적합하지 않았는지 검증한다.
- 모델 결과를 현장 위험 확정 판정으로 표현하지 않는다.
